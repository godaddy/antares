# Drawer Primitive

The `Drawer` primitive is a component that creates an expandable panel. It is meant to hold content and reveal it after some interaction.

## Purpose

- Support sidebar navigation, where the drawer can be collapsed to show minimal navigation items and expand to show the full navigation.
- Support bottom sheet functionality, where the drawer can be expanded to show more options or content.
- Support disclosures, where the drawer can be expanded to show more information or content inline within a composition.

> [!NOTE]
> The drawer _could_ also support a modal-like behavior, where the drawer is expanded from the center of the parent container.
> See the [Composition](#composition) section below for more information. This would require a component name change to indicate that this component handles this sort of behavior.

## Primitive Composition

We may explore using the Disclosure resources from React Aria, either the [component](https://react-spectrum.adobe.com/react-aria/Disclosure.html) or the [hook](https://react-spectrum.adobe.com/react-aria/useDisclosure.html). Though it's implementation may not aligned to our expectations below. Specifically the ability to ship the drawer without a trigger.

## Props

### placement

From the [`useOverlayPosition` hook](https://react-spectrum.adobe.com/react-aria/useOverlayPosition.html), the placement of the element with respect to its positioning parent. This will accept strings such as `top`, `bottom`, `left`, `right`. The use of left and right will flip in RTL contexts so that the implementation is consistent. See the [Composition](#composition) section for more information on other requirements for this to work.

### isExpanded

From the [`useDisclosure` hook](https://react-spectrum.adobe.com/react-aria/useDisclosure.html), whether the drawer is expanded or collapsed.
If the `animate` prop is provided, the drawer will animate expanded or collapsed based on this flag.

### defaultExpanded

From the [`useDisclosure` hook](https://react-spectrum.adobe.com/react-aria/useDisclosure.html), the `defaultExpanded` prop is a flag that determines if the drawer is expanded or collapsed by default.
If not provided, the drawer will be collapsed by default.

### onExpandedChange

From the [`useDisclosure` hook](https://react-spectrum.adobe.com/react-aria/useDisclosure.html), this is a callback that is fired when the drawer is expanded or collapsed.

### shouldCloseOnBlur

From the [`useOverlay` hook](https://react-spectrum.adobe.com/react-aria/useOverlay.html), whether the overlay should close when focus is lost or moves outside it.

### isKeyboardDismissDisabled

From the [`useOverlay` hook](https://react-spectrum.adobe.com/react-aria/useOverlay.html), whether pressing the escape key to close the overlay should be disabled.

### shouldCloseOnInteractOutside

From the [`useOverlay` hook](https://react-spectrum.adobe.com/react-aria/useOverlay.html), when user interacts with the argument element outside of the overlay ref, to return true if onClose should be called. This gives you a chance to filter out interaction with elements that should not dismiss the overlay. By default, onExpandedChange will always be called on interaction outside the overlay ref.

```js
<Drawer shouldCloseOnInteractOutside={ (element) => return mightClose(element) } />
```

> [!NOTE]
> The naming convention here coming from the hooks is a bit random in my opinion.
> If we're able to deviate, I'd rather align these to use the words "should" and "dismiss" along with some other more descriptive names.
> - `shouldCloseOnBlur` -> `shouldDismissOnBlur`
> - `isKeyboardDismissDisabled` -> `shouldDismissOnEscape`
> - `shouldCloseOnInteractOutside` -> `shouldDismissOnElementInteract`

### maxWidth/maxHeight

The maximum width and height of the drawer in CSS units. Percentage is based on the parent.
When the drawer is **expanded**, the drawer will be constrained to these values.
If not provided, when the drawer is expanded it will be the size of its contents.

### minWidth/minHeight

The minimum width and height of the drawer in CSS units. Percentage is based on the parent.
When the drawer is **collapsed**, the drawer will be constrained to these values.
This allows for the possibility of the drawer showing a small amount of content when collapsed such as a button to expand the drawer.
If not provided, when the drawer is collapsed these values will be `0`, causing the drawer to be visually hidden.

> [!NOTE]
> If the drawer is an extension of the layout components then these props will be provided from the layout component.

### isTouch

The `isTouch` prop is a flag that determines if the drawer should allow touch gestures.
This will also include a handle on edges opposite of the attachment edge to indicate where the drawer can be swiped.

```js
<Drawer isTouch={ window.matchMedia('(pointer: coarse) and (hover: none)').matches } />
```

- Swiping a minimized drawer away from it's attached edge will expand the drawer.
- Swiping a maximized drawer toward it's attached edge will collapse the drawer.

### animate

The `animate` prop determines if and/or how the drawer should animate when it is expanded or collapsed from its minimum size to its maximum size.
If provided as the flag `true`, the drawer will animate with default settings.
Otherwise, you may provide a string that represents a CSS transition, specifically with the timing function and duration.

```js
<Drawer animate="0.3s ease-in-out" />
```

> [!NOTE]
> Due to the examples provided by Adobe below, it is possible that we avoid the use of `animate` and instead use Framer Motion to animate the drawer. This is an open question.

## Vaul

An existing implementation of a drawer is [Vaul](https://vaul.emilkowal.ski/). One of the features of this library is [snap points](https://vaul.emilkowal.ski/docs/features/snap-points). These indicate where the drawer will snap to when it is opened.

```js
const snapPoints = ['148px', '355px', 1];
const [snap, setSnap] = useState(snapPoints[0]);
<Drawer.Root snapPoints={snapPoints} activeSnapPoint={snap} setActiveSnapPoint={setSnap}>
  <Drawer.Trigger>
    <Button>Open Drawer</Button>
  </Drawer.Trigger>
  <Drawer.Portal>
    <Drawer.Overlay />
    <Drawer.Content>
      <Drawer.Close>Close Drawer</Drawer.Close>
    </Drawer.Content>
  </Drawer.Portal>
</Drawer.Root>
```

Vaul only supplies components and does not provide any hooks. It might be more beneficial to have hooks available to support the components instead. So we may consider acquiring [the source code](https://github.com/emilkowalski/vaul/blob/main/src/use-snap-points.ts) and using it as a reference for our own hooks.

Vaul provided other features such as [adjusting the placement when a mobile keyboard is displayed](https://github.com/emilkowalski/vaul/blob/main/src/index.tsx#L477-L533) and [nesting drawers](https://github.com/emilkowalski/vaul/blob/main/src/index.tsx#L681-L706).

## Accessibility

- If the drawer is meant to disrupt the user from other tasks, then it should be given a `role="dialog"` and `aria-modal="true"`. Otherwise, it should be given a `role="region"` unless it is provided a more semantic HTML element (eg., `<nav/>`).
- `aria-expanded` will be used to indicate the expanded state of the drawer.
- If the drawer is meant to be completely hidden, then it should be given a `aria-hidden="true"` or consider [`hidden="until-found"`](https://developer.mozilla.org/en-US/docs/Web/HTML/Reference/Global_attributes/hidden#the_hidden_until_found_state).
- We may consider making the use of `id` required such that it can be given a connection to a `aria-controls` attribute. Or we could consider some fun slot shenanigans to make it work with `useId()` (maybe?).

```jsx
<Drawer id="drawer">
  <Button aria-controls="drawer">
    Close Drawer
  </Button>
</Drawer>
```

## Focus Trapping

If the composition is expecting focus to be trapped within the drawer,
then a [`<FocusScope/>`](https://react-spectrum.adobe.com/react-aria/FocusScope.html) must be provided.

```js
<Drawer>
  <FocusScope>
    <input type="text" />
  </FocusScope>
</Drawer>
```

## Composition

Specifically in the case of potentionally replacing Application Sidebar, we'll need the ability for the drawer to change between affecting sibling content and appearing above it as a configuration. In React Aria, this is handled by providing a `<Disclosure>` as the parent and a `<DisclosurePanel>` as the panel. Other children to the parent are siblings to the panel.

```jsx
<Disclosure>
  <Heading>
    <Button slot="trigger">
      <ChevronRight size={18} />
      System Requirements
    </Button>
  </Heading>
  <DisclosurePanel>
    <p>Details about system requirements here.</p>
  </DisclosurePanel>
</Disclosure>
```

We could adopt this pattern directly with a different naming convention:

```jsx
import { Drawer, DrawerParent } from '@bento/drawer';

function Chrome({ children }) {
  const [isExpanded, setIsExpanded] = useState(false);
  return (
    <DrawerParent>
      <Drawer placement="left" isExpanded={ isExpanded }>
        <Button onClick={ () => setIsExpanded(!isExpanded) }>Toggle Drawer</Button>
      </Drawer>
      { children }
    </DrawerParent>
  )
}
```

The `DrawerParent` component could then be given a `portalContainer` ([name from React Aria](https://github.com/adobe/react-spectrum/blob/main/packages/%40react-aria/overlays/src/Overlay.tsx#L26)) prop to indicate that the parent is to be portaled. The parent should also include all of the relevant styles to support the drawer in either inline or overlay modes.

```jsx
<DrawerParent portalContainer={ document.body }>
  <Drawer placement="bottom" isExpanded={ isExpanded }>
    <Button onClick={ () => setIsExpanded(false) }>Close Drawer</Button>
  </Drawer>
  { children }
</DrawerParent>
```
