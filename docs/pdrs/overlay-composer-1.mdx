# Overlay Primitive PDR

## Purpose

The Overlay primitive provides a foundational building block for creating layered UI experiences where content appears above the main application view. It serves as a blank canvas that enables teams to build custom modal and drawer experiences with full flexibility, particularly optimized for mobile-first patterns.

The Overlay primitive addresses several key scenarios:

- Displaying content that requires immediate user attention or action
- Creating focused interaction spaces that temporarily obscure the main content
- Supporting mobile-first patterns like drawers, sheets, and action menus
- Enabling rich compositional patterns through layering of UI elements

This primitive will typically be consumed by:
- Application developers building modal interfaces or drawers
- Component authors creating specialized dialog components
- Teams building loading indicators or notification systems
- Any UI pattern requiring a visual layer between content and application

### Unique Attributes

The Overlay primitive differs from similar components in these key ways:

- **Pure foundation**: Unlike most dialog components that combine multiple concerns, the Overlay provides only the backdrop and layering functionality, allowing maximum composition flexibility
- **React Aria integration**: Built directly on React Aria's `useModalOverlay` and `useOverlay` hooks for best-in-class accessibility
- **Slot-based composition**: Follows Bento's composition guidelines, passing state through slots for maximum customization
- **Mobile-optimized**: Designed with mobile-first patterns in mind, supporting touch interactions and mobile gestures

## Primitive Composition

The Overlay primitive composes several standalone reusable primitives:

### Core Primitives

1. **@bento/portal** - Handles DOM rendering placement outside the normal hierarchy
2. **@bento/backdrop** - Provides the semi-transparent backdrop layer behind overlay content
3. **@bento/overlay-stack** - Manages multiple overlays with proper z-index and accessibility
4. **@bento/focus-lock** - Contains keyboard focus within the overlay (existing primitive)
5. **@bento/scroll-lock** - Prevents background scrolling (existing primitive)
6. **@bento/dismiss-button** - Provides accessible dismiss functionality for screen readers
7. **@bento/container** - Polymorphic rendering for overlay content

### React Aria Integration

The Overlay primitive builds directly upon React Aria's accessibility-focused hooks:

- **useModalOverlay**: For backdrop and modal behavior (combines `useModal` + `useOverlay`)
- **useOverlay**: For core overlay behavior (dismissal, escape key, outside clicks)
- **useDialog**: For dialog-specific accessibility (when used in dialog contexts)

## Internal Structure & Reuse Potential

The Overlay component is designed as a minimal foundation that delegates to specialized primitives:

```tsx
import { useModalOverlay } from '@react-aria/overlays';
import { useDialog } from '@react-aria/dialog';
import { useOverlayState } from '@react-stately/overlays';
import { useProps } from '@bento/use-props';
import { useDataAttributes } from '@bento/use-data-attributes';
import { withSlots } from '@bento/slots';
import { Container } from '@bento/container';
import { Portal } from '@bento/portal';
import { Backdrop } from '@bento/backdrop';
import { useOverlayStack } from '@bento/overlay-stack';
import { FocusLock } from '@bento/focus-lock';
import { ScrollLock } from '@bento/scroll-lock';
import { DismissButton } from '@bento/dismiss-button';
import { useId, useEffect } from 'react';

export const Overlay = withSlots('BentoOverlay', function Overlay(args) {
  const { props: processedProps } = useProps(args, {
    isOpen: state.isOpen,
    isDismissable: state.isDismissable,
    isModal: state.isModal,
    overlayType: 'overlay' // or 'modal', 'popover', etc.
  });

  const state = useOverlayState(processedProps);
  const id = useId();
  const stack = useOverlayStack();
  const ref = useRef(null);
  const { modalProps, underlayProps } = useModalOverlay(processedProps, state, ref);
  const { dialogProps } = useDialog(processedProps, ref);

  // Register with overlay stack when open
  const overlayInfo = state.isOpen
    ? stack.register(id, processedProps.overlayType || 'overlay')
    : null;

  useEffect(() => {
    if (!state.isOpen) {
      stack.unregister(id);
    }
    return () => stack.unregister(id);
  }, [state.isOpen, id, stack]);

  const dataAttributes = useDataAttributes({
    open: state.isOpen,
    modal: state.isModal,
    dismissable: state.isDismissable,
    'overlay-id': id,
    'overlay-type': processedProps.overlayType,
    'overlay-topmost': overlayInfo?.isTopmost
  });

  // Don't render if not open
  if (!state.isOpen || !overlayInfo) return null;

  // Only topmost modal should have aria-modal="true"
  const shouldBeModal = state.isModal && overlayInfo.isTopmost;

  // Get layer orchestration if mfeId provided (for cross-MFE coordination)
  const layerOrchestration = processedProps.mfeId
    ? useLayerOrchestration({
        type: processedProps.overlayType || 'overlay',
        priority: processedProps.priority || 2,
        mfeId: processedProps.mfeId,
        isOpen: state.isOpen,
        controlled: processedProps.controlled
      })
    : null;

  // Use orchestrated z-index if available, otherwise use stack z-index
  const finalZIndex = layerOrchestration?.zIndex ?? overlayInfo?.zIndex ?? 1000;

  return (
    <Portal>
      <Backdrop
        {...underlayProps}
        {...dataAttributes}
        style={{
          zIndex: finalZIndex - 1,
          ...underlayProps.style
        }}
      />
      <FocusLock contain restoreFocus autoFocus>
        <ScrollLock isDisabled={!state.isOpen}>
          <Container
            {...modalProps}
            {...dialogProps}
            {...dataAttributes}
            slot="content"
            slots={processedProps.slots}
            style={{
              zIndex: finalZIndex,
              '--layer-z-index': finalZIndex, // CSS variable for child layers
              ...processedProps.style
            }}
            aria-modal={shouldBeModal ? 'true' : undefined}
            aria-hidden={!overlayInfo?.isTopmost ? 'true' : undefined}
          >
            {processedProps.children}
            <DismissButton onDismiss={() => state.close()} />
          </Container>
        </ScrollLock>
      </FocusLock>
    </Portal>
  );
});
```

**Reuse potential:**
- Modals and dialogs
- Drawers and bottom sheets
- Popovers and tooltips
- Toast notifications
- Loading overlays
- Fullscreen navigation menus

## React Aria or External Hook Integration

### Why React Aria?

After extensive research of different overlay implementations, React Aria stands out as the optimal foundation:

1. **Accessibility excellence**: React Aria is built from the ground up with accessibility as the primary focus, ensuring WCAG compliance
2. **Comprehensive feature set**: Provides complete solution for overlay functionality:
   - Focus management with `useFocusScope` (wrapped in our FocusLock)
   - Scroll locking with `usePreventScroll` (wrapped in our ScrollLock)
   - Backdrop interaction with `useModalOverlay`
   - Trigger management with `useOverlayTrigger`
3. **Battle-tested**: Used extensively in Adobe's product ecosystem
4. **Excellent mobile support**: Handles complex mobile edge cases including iOS Safari
5. **Active maintenance**: Regular updates ensure compatibility with browser changes

### Comparative Analysis

We evaluated several libraries for overlay implementation:

| Library | Strengths | Weaknesses | Focus Management | Multiple Overlay Support |
|--------|-----------|------------|------------------|-------------------------|
| React Aria | Exceptional a11y, robust focus management, strong mobile support | Less composable API | FocusScope component with complete trapping | OverlayContainer system |
| Radix UI | Highly composable API, excellent animation support | Some mobile edge cases | Dialog-specific focus handling | Built-in dialog stack |
| Ark UI | Flexible API, excellent composition | More complex API | FocusTrap component | Store-based dialog state |
| Headless UI | Simplicity, good animation integration | Less flexible | Dialog-specific focus handling | Limited |
| Material UI | Performance optimizations, comprehensive features | Less headless, more opinionated | Uses custom FocusTrap | ModalManager utility |

React Aria provides the strongest foundation for accessibility and mobile support. We enhance it with Bento's composition model and slot-based customization.

## Architecture & Features

### Props Interface

```tsx
interface OverlayProps extends Slots {
  /**
   * Whether the overlay is open.
   */
  isOpen?: boolean;

  /**
   * Default open state (uncontrolled).
   */
  defaultOpen?: boolean;

  /**
   * Callback fired when open state changes.
   */
  onOpenChange?: (isOpen: boolean) => void;

  /**
   * Whether the overlay is modal (blocks interaction with background).
   */
  isModal?: boolean;

  /**
   * Whether the overlay can be dismissed.
   */
  isDismissable?: boolean;

  /**
   * Whether pressing Escape dismisses the overlay.
   */
  isKeyboardDismissDisabled?: boolean;

  /**
   * Whether clicking outside dismisses the overlay.
   */
  shouldCloseOnInteractOutside?: boolean | ((element: Element) => boolean);

  /**
   * Whether the overlay closes on blur.
   */
  shouldCloseOnBlur?: boolean;

  /**
   * Type of overlay for stacking purposes.
   * Affects base z-index and stacking behavior.
   * @default 'overlay'
   */
  overlayType?: 'tooltip' | 'popover' | 'dropdown' | 'overlay' | 'modal' | 'toast' | 'alert';

  /**
   * Content to render inside the overlay.
   */
  children?: React.ReactNode;
}
```

## Accessibility Highlights

The Overlay primitive inherits React Aria's excellent accessibility features:

- **ARIA roles**: Automatically applies `role="dialog"` or `role="alertdialog"` when appropriate
- **Modal state**: Uses `aria-modal="true"` for modal overlays
- **Focus management**: Implements proper focus trapping via FocusLock primitive
- **Keyboard support**:
  - Escape key to dismiss (configurable)
  - Tab navigation contained within the overlay
  - Return focus to trigger element on close
- **Screen reader support**:
  - Hides background content from screen readers when modal
  - DismissButton provides accessible dismissal mechanism
- **Proper handling of focus order** with multiple overlays

For mobile accessibility:
- Touch targets follow accessibility guidelines (minimum 44×44px)
- Screen reader gestures properly supported
- Visible focus indicators maintained for keyboard users

## Internationalization, RTL, and Mobile Considerations

The Overlay primitive is designed to work across different languages and layouts:

- **RTL support**: The overlay positioning and animations work correctly in right-to-left layouts
- **Mobile considerations**:
  - Larger touch targets for mobile interactions
  - Special handling for iOS Safari to prevent background content scrolling
  - Support for mobile gestures (swipe to dismiss via composition)
  - Adjustments for virtual keyboard appearance
  - Responsive sizing and positioning

- **Internationalization**: The component handles text expansion from different languages by using flexible layouts

## Competitive Research

### React Aria

React Aria provides a robust set of hooks for building accessible overlays:

- **useOverlay**: Manages ARIA attributes and keyboard interactions
- **useModalOverlay**: Adds modal-specific behaviors like backdrop interaction
- **useFocusScope**: Handles focus containment
- **usePreventScroll**: Manages scroll locking with excellent cross-browser support
- **useOverlayPosition**: Supports positioning relative to trigger elements
- **useOverlayTrigger**: Connects trigger elements to overlays

React Aria excels at accessibility but has a more imperative API that requires additional work to fit into Bento's composable component model.

### Park UI / Ark UI

Park UI (built on Ark UI) offers a highly composable dialog component:

```jsx
// Ark UI approach
<Dialog.Root>
  <Dialog.Trigger />
  <Dialog.Portal>
    <Dialog.Backdrop />
    <Dialog.Content>
      <Dialog.Title />
      <Dialog.Close />
    </Dialog.Content>
  </Dialog.Portal>
</Dialog.Root>
```

Strengths:
- Excellent composition model
- Good animation support via data attributes
- Solid accessibility

Weaknesses:
- Some mobile edge cases
- Different composition model (asChild pattern)

### Radix UI

Radix UI offers a highly composable dialog component:

```jsx
// Radix UI approach
<Dialog.Root>
  <Dialog.Trigger />
  <Dialog.Portal>
    <Dialog.Overlay />
    <Dialog.Content>
      <Dialog.Title />
      <Dialog.Description />
      <Dialog.Close />
    </Dialog.Content>
  </Dialog.Portal>
</Dialog.Root>
```

Strengths:
- Excellent composition model
- Good animation support via data attributes
- Solid accessibility

Weaknesses:
- Some mobile edge cases
- Occasional focus management issues in complex scenarios

### Multiple overlays (stacking)

```tsx
import { OverlayStack } from '@bento/overlay-stack';

import { Button } from '@bento/button';
import { useState } from 'react';

function MultipleOverlaysExample() {

        isModal

          <Text>This is the first modal in the stack.</Text>

          </Button>
          <Button onPress={() => setFirstOpen(false)}>Close</Button>
        </Container>
      </Overlay>

      {/* Second modal stacks above first */}
      <Overlay
        isOpen={secondOpen}
        onOpenChange={setSecondOpen}
        overlayType="modal"
        isModal
      >
        <Container slot="content" className="modal">
          <Text as="h2">Second Modal</Text>
          <Text>
            This modal appears above the first one.
            Only this one has aria-modal="true".
          </Text>
          <Button onPress={() => setSecondOpen(false)}>Close</Button>
        </Container>
      </Overlay>
    </OverlayStack.Provider>
  );
}
```

### Micro-frontend coordination (Layer Orchestrator)

**Zero configuration needed!** Components handle z-index coordination automatically:

```tsx
import { Overlay } from '@bento/overlay';
import { LAYER_PRIORITY } from '@bento/layer-orchestrator';
import { useState } from 'react';

// Example: Cookie Consent MFE (Bento v1)
function CookieConsentMFE() {
  const [isOpen, setIsOpen] = useState(true);

  // Just use the component - z-index coordination is automatic!
  return (
    <Overlay
      isOpen={isOpen}
      onOpenChange={setIsOpen}
      isModal
      mfeId="legal-mfe"
      type="modal"
      priority={LAYER_PRIORITY.HIGH} // High priority - interrupts tours
      controlled // Only render when topmost
    >
      <Container slot="content">
        <Text as="h2">Cookie Consent</Text>
        <Text>We need your consent to use cookies.</Text>
        <Button onPress={() => setIsOpen(false)}>Accept</Button>
      </Container>
    </Overlay>
  );
}

// Example: Product Tour MFE (Bento v2)
function ProductTourMFE() {
  const [currentStep, setCurrentStep] = useState(0);
  const steps = [/* ... */];

  // Just use the component - queuing and z-index are automatic!
  return (
    <Overlay
      isOpen={currentStep < steps.length}
      overlayType="tour"
      mfeId="product-mfe"
      priority={LAYER_PRIORITY.LOW} // Low priority - can be interrupted
      controlled // Only render when topmost (queued otherwise)
    >
      <Container slot="content">
        <Text>{steps[currentStep].content}</Text>
        <Button onPress={() => setCurrentStep(currentStep + 1)}>Next</Button>

```

**Key points:**
- **No manual z-index**: Components automatically set CSS variables and handle calc()
- **No useLayerOrchestration hook**: Components use it internally
- Each MFE provides unique `mfeId` for identification
- `priority` determines which overlay appears above others
- `controlled` prop handles queuing automatically
- All MFEs independently calculate z-index from synced state
- No host application or leader required

See `layer-orchestrator.mdx` for detailed implementation and architecture.

## Multiple Overlays and Stacking

### Single Bundle: OverlayStack

When multiple overlays are open simultaneously **within the same JavaScript bundle**, use `OverlayStack.Provider` to manage proper stacking:

1. **Visual stacking**: Higher z-index for overlays opened later
2. **Accessibility**: Only topmost modal has `aria-modal="true"`
3. **Focus management**: Focus transfers correctly when overlays open/close
4. **Type-based layering**: Different overlay types have different base z-indexes

**Use OverlayStack when:**
- All overlays come from the same JavaScript bundle
- Shared React context is available
- No micro-frontend architecture

See `overlay-stack.mdx` for detailed stacking implementation.

### Micro-Frontends: Layer Orchestrator

When overlays come from **different JavaScript bundles (micro-frontends)**, use the **Layer Orchestrator** for distributed, peer-to-peer coordination **without requiring a host application**:

1. **Distributed coordination**: Each MFE maintains its own state and syncs via events (no single leader)
2. **Event-driven synchronization**: All instances broadcast state changes; all instances update independently
3. **Version compatibility**: Different MFEs can use different Bento versions (v1, v2, v3) via protocol versioning
4. **Cross-bundle coordination**: Works via DOM/CustomEvents and optional DOM/sessionStorage backup
5. **Priority-based control**: Higher priority layers automatically appear above lower priority layers
6. **Queue system**: Experiences can be queued when higher priority experiences are active
7. **Universal orchestration**: Works for overlays, toasts, product tours, any layered UI
8. **No single point of failure**: If one MFE fails, others continue coordinating independently

**Use Layer Orchestrator when:**
- Multiple micro-frontends launch competing overlay experiences
- Different JavaScript bundles cannot share React context
- No host application available to coordinate
- MFEs may be using different Bento versions (v1, v2, v3)
- Need to coordinate cookie consent, product tours, live chat, modals from different MFEs
- Need priority-based control (e.g., cookie consent interrupts product tour)
- Insert order/z-index alone don't work (MFEs load in unpredictable order)

**Example scenarios:**
- Product tour MFE (Bento v2) starts overlay, Legal MFE (Bento v1) shows cookie consent (high priority) → Cookie consent appears above tour via distributed z-index calculation
- Live chat MFE (Bento v3) active, another MFE (Bento v1) needs critical modal → Modal appears above chat via event-based coordination
- Two MFEs (different versions) want to show growls → Only one visible at a time, others queued via distributed queue

**Key difference from OverlayStack**: OverlayStack requires shared React context (single bundle). Layer Orchestrator uses distributed event-driven synchronization across bundles and versions without requiring a leader or host.

See `layer-orchestrator.mdx` for detailed peer-to-peer orchestration implementation.

### Stacking Behavior (Both Systems)

- **Registration**: Overlays register when they open
- **Z-index calculation**: Based on overlay type and existing stack/orchestrator state
- **Topmost determination**: Only topmost overlay is fully accessible
- **Focus transfer**: When overlay closes, focus returns to previous overlay or trigger

### UX Considerations

While multiple overlays are technically supported, consider UX:

- **Confirmation dialogs**: Appropriate use case
- **Nested workflows**: Can work but consider alternatives
- **Too many layers**: Can confuse users; prefer alternatives like:
  - Inline expansion
  - Side panels
  - Step-by-step within single overlay
  - Navigation to dedicated page

## Decision Checklist

When building with the Overlay primitive:

- [ ] Is this overlay modal or non-modal?
- [ ] Does it need a backdrop?
- [ ] Should it close on outside click?
- [ ] Should it close on Escape key?
- [ ] Does it need focus trapping? (usually yes for modals)
- [ ] Does it need scroll locking? (usually yes for modals)
- [ ] How should it be positioned? (centered, drawer, etc.)
- [ ] What content needs to be in the overlay?
- [ ] Are there accessibility requirements specific to this use case?
- [ ] Will this overlay appear with others? (requires OverlayStack.Provider)
- [ ] What overlay type is this? (modal, popover, tooltip, etc.)
