# Overlay Primitive PDR

## Purpose

The Overlay primitive provides a foundational building block for creating layered UI experiences where content appears above the main application view. It serves as a blank canvas that teams can use to build modals, drawers, popovers, tooltips, and other overlay-based patterns.

The Overlay primitive addresses several key scenarios:

- Displaying content that requires immediate user attention or action
- Creating focused interaction spaces that temporarily obscure or highlight main content
- Supporting mobile-first patterns like drawers, sheets, and bottom panels
- Enabling rich compositional patterns through layering of UI elements
- Managing multiple simultaneous overlays with proper stacking and focus management

This primitive will typically be consumed by:
- Application developers building modal interfaces or drawers
- Component authors creating specialized dialog components
- Teams building loading indicators or notification systems
- Any UI pattern requiring a visual layer between content and application

### Unique Attributes

The Overlay primitive differs from similar components in these key ways:

- **Pure foundation**: Unlike most dialog components that combine multiple concerns, the Overlay provides only the backdrop, focus management, and dismissal behavior, allowing maximum composition flexibility
- **Slot-based composition**: Follows Bento's slot system - consumers provide components as children, overlay passes state/handlers via slots
- **Portal rendering**: Automatically renders content outside the DOM hierarchy via a Portal primitive
- **Focus management**: Integrated focus trapping capabilities via FocusLock primitive that follow WAI-ARIA best practices
- **Scroll locking**: Cross-browser scroll management that prevents background content scrolling
- **Overlay stacking**: Built-in support for managing multiple simultaneous overlays with proper z-index and focus management
- **Accessibility-first**: Built on React Aria's `useModalOverlay` and `useOverlay` hooks, ensuring WCAG compliance

## Primitive Composition

The Overlay primitive composes several standalone reusable primitives, maximizing code sharing across the Bento ecosystem:

### Core Primitive Packages

Each primitive is its own package following Bento's architecture:

- **@bento/portal**: Handles DOM rendering placement (see `portal-primitive.mdx`)
- **@bento/focus-lock**: Contains keyboard focus within a boundary (already documented in `focus-lock-primtive.mdx`)
- **@bento/scroll-lock**: Prevents background scrolling (see `scroll-lock-primitive.mdx`)
- **@bento/backdrop**: Provides visual backdrop that sits behind content (see `backdrop-primitive.mdx`)
- **@bento/dismiss-button**: Accessible dismiss functionality (see `dismiss-button-primitive.mdx`)
- **@bento/overlay-stack**: Manages multiple simultaneous overlays (see `overlay-stack-primitive.mdx`)
- **@bento/container**: Base primitive for polymorphic rendering

### Overlay Component Structure

The Overlay is a single component that accepts children with slot attributes. Consumers provide the actual components (Button, Backdrop, etc.) and Overlay passes state/handlers via slots:

```tsx
<Overlay>
  <Button slot="trigger">Open</Button>
  <Backdrop slot="backdrop" />
  <Container slot="content">
    <Text>Overlay content</Text>
    <DismissButton slot="close">Close</DismissButton>
  </Container>
</Overlay>
```

The Overlay component:
- Manages state (open/close)
- Passes trigger props via `slot="trigger"`
- Passes backdrop props via `slot="backdrop"`
- Passes content props via `slot="content"`
- Passes close handler via `slot="close"`

### React Aria Integration

All primitives build directly upon React Aria's accessibility-focused hooks:

- **useModalOverlay**: For backdrop and modal behavior
- **useOverlay**: For core overlay behavior, dismissal, and accessibility
- **useOverlayTrigger**: For trigger elements (props passed via slots)
- **usePreventScroll**: For scroll locking (used via ScrollLock primitive)
- **FocusScope**: For focus containment (used via FocusLock primitive)

## Internal Structure & Reuse Potential

The Overlay component manages state and coordinates all child components via slots:

```tsx
import React, { useRef } from 'react';
import { useOverlayTriggerState } from '@react-stately/overlays';
import { useModalOverlay, useOverlay, useOverlayTrigger } from '@react-aria/overlays';
import { useProps } from '@bento/use-props';
import { useDataAttributes } from '@bento/use-data-attributes';
import { Container } from '@bento/container';
import { withSlots, Slots } from '@bento/slots';
import { Portal } from '@bento/portal';
import { FocusLock } from '@bento/focus-lock';
import { ScrollLock } from '@bento/scroll-lock';
import { useOrchestratedLayer } from '@bento/overlay-stack';

export interface OverlayProps extends Slots {
  isOpen?: boolean;
  defaultOpen?: boolean;
  onOpenChange?: (isOpen: boolean) => void;
  isDismissable?: boolean;
  isKeyboardDismissDisabled?: boolean;
  isModal?: boolean;
  /**
   * Priority for orchestration (when multiple MFEs compete)
   */
  priority?: 'critical' | 'important' | 'normal' | 'low';
  /**
   * Bundle/MFE identifier for orchestration
   */
  bundleId?: string;
  children?: React.ReactNode;
}

export const Overlay = withSlots('BentoOverlay', function Overlay(args: OverlayProps) {
  const state = useOverlayTriggerState({
    isOpen: args.isOpen,
    defaultOpen: args.defaultOpen,
    onOpenChange: args.onOpenChange,
    isDismissable: args.isDismissable,
    isKeyboardDismissDisabled: args.isKeyboardDismissDisabled,
  });

  const overlayRef = useRef<HTMLDivElement>(null);
  const triggerRef = useRef<HTMLElement>(null);

  const { modalProps, underlayProps } = useModalOverlay(
    {
      isDismissable: args.isDismissable,
      isKeyboardDismissDisabled: args.isKeyboardDismissDisabled,
      isModal: args.isModal ?? true,
    },
    state,
    overlayRef
  );

  const { overlayProps } = useOverlay(
    {
      onClose: () => state.setOpen(false),
      isDismissable: args.isDismissable,
      isKeyboardDismissDisabled: args.isKeyboardDismissDisabled,
    },
    overlayRef
  );

  const { triggerProps } = useOverlayTrigger(
    { type: 'dialog' },
    state,
    triggerRef
  );

  // Register with orchestration system for cross-bundle z-index management
  const orchestration = useOrchestratedLayer({
    type: args.isModal ? 'modal' : 'overlay',
    priority: args.priority,
    isOpen: state.isOpen,
    bundleId: args.bundleId,
  });

  const { ref: orchestrationRef, zIndex, stackIndex, isTopmost } = orchestration;

  // Attach orchestration ref to overlay element
  if (overlayRef.current && orchestrationRef.current !== overlayRef.current) {
    orchestrationRef.current = overlayRef.current;
  }

  const { props: processedProps } = useProps(args, {
    isOpen: state.isOpen,
    isModal: args.isModal ?? true,
    zIndex,
    stackIndex,
    isTopmost,
  });

  const dataAttributes = useDataAttributes({
    open: state.isOpen,
    modal: args.isModal ?? true,
    stackIndex,
    topmost: isTopmost,
  });

  if (!state.isOpen) {
    // When closed, only render trigger if provided
    return processedProps.children ? (
      <Container
        slot="root"
        slots={{
          trigger: {
            ...triggerProps,
            ref: triggerRef,
            onClick: () => state.setOpen(true),
          },
        }}
      >
        {processedProps.children}
      </Container>
    ) : null;
  }

  return (
    <Portal>
      <FocusLock contain restoreFocus autoFocus>
        <ScrollLock>
          <Container
            {...processedProps}
            {...modalProps}
            {...overlayProps}
            {...dataAttributes}
            ref={overlayRef}
            slot="root"
            style={{
              // Set CSS variable for nested layers to inherit
              '--layer-z-index': String(zIndex),
              zIndex: 'var(--layer-z-index)',
              ...processedProps.style,
            }}
            slots={{
              // Pass trigger props to slot="trigger" children
              trigger: {
                ...triggerProps,
                ref: triggerRef,
                onClick: () => state.setOpen(true),
              },
              // Pass backdrop props to slot="backdrop" children
              backdrop: {
                ...underlayProps,
                onClick: args.isDismissable ? () => state.setOpen(false) : undefined,
              },
              // Pass content props to slot="content" children
              content: {
                ...modalProps,
                ...overlayProps,
                role: 'dialog',
                'aria-modal': args.isModal && isTopmost ? 'true' : undefined,
                'aria-hidden': !isTopmost ? 'true' : undefined,
              },
              // Pass close handler to slot="close" children
              close: {
                onDismiss: () => state.setOpen(false),
              },
            }}
          >
            {processedProps.children}
          </Container>
        </ScrollLock>
      </FocusLock>
    </Portal>
  );
});
```

**Reusable across:**
- Modals and dialogs
- Drawers and side panels
- Popovers
- Toast notifications
- Any overlay-based pattern

## React Aria or External Hook Integration

### Why React Aria?

After extensive research of different overlay implementations, React Aria stands out as the optimal foundation:

1. **Accessibility excellence**: React Aria is built from the ground up with accessibility as the primary focus, ensuring WCAG compliance

2. **Comprehensive feature set**: React Aria provides a complete solution for overlay functionality:
   - Focus management with `FocusScope` component
   - Scroll locking with `usePreventScroll`
   - Backdrop interaction with `useModalOverlay`
   - Trigger management with `useOverlayTrigger`

3. **Battle-tested in production**: Used extensively in Adobe's product ecosystem, ensuring robust performance in real-world applications

4. **Excellent mobile support**: Handles complex mobile edge cases that other libraries miss, including iOS Safari scroll handling

5. **Active maintenance**: Regular updates and community support ensure the library stays current with browser changes and accessibility standards

### Hooks Used

- **useOverlayTriggerState**: State management for overlay open/close state from @react-stately/overlays
- **useModalOverlay**: Provides modal-specific props and accessibility attributes
- **useOverlay**: Core overlay behavior (dismissal, keyboard handling)
- **useOverlayTrigger**: Connects trigger elements to overlays (props passed via slots)
- **usePreventScroll**: Prevents background scrolling (wrapped in ScrollLock primitive)

No additional logic is needed beyond React Aria's implementation - it handles all edge cases for overlay behavior across devices.

## Architecture & Features

### State Management

The Overlay uses React Aria's `useOverlayTriggerState` for state management, supporting both controlled and uncontrolled modes:

- **Controlled**: `isOpen` and `onOpenChange` props
- **Uncontrolled**: `defaultOpen` prop

### Dismissal Behavior

The overlay supports multiple dismissal patterns:

- **Click outside**: Dismissable by clicking the backdrop (controlled by `isDismissable`)
- **Escape key**: Keyboard dismissal (controlled by `isKeyboardDismissDisabled`)
- **Programmatic**: Can be closed via state updates or DismissButton with `slot="close"`

### Modal vs Non-Modal

The overlay can operate in two modes:

- **Modal** (`isModal={true}`): Blocks interaction with background content, uses `aria-modal="true"` (only when topmost)
- **Non-Modal** (`isModal={false}`): Does not block background interaction, useful for tooltips and popovers

### Portal Rendering

All overlay content is rendered via the Portal primitive, ensuring proper z-index stacking and avoiding DOM hierarchy issues.

### Focus Management

Focus is managed through the FocusLock primitive which:
- Traps focus within the overlay when open (only topmost when stacked)
- Returns focus to the trigger element when closed
- Handles initial focus placement
- Manages focus order for accessibility

### Scroll Locking

Background scrolling is prevented via the ScrollLock primitive, which:
- Prevents body scrolling when overlay is open
- Handles iOS Safari edge cases
- Maintains scroll position for restoration

## Overlay Stacking Support

A critical requirement is handling multiple simultaneous overlays (modals, drawers, popovers) **across multiple micro front-end applications**. The Overlay primitive integrates with `@bento/overlay-stack` to provide **peer-to-peer cross-bundle orchestration**.

**Important**: There is **no host application** required - MFEs coordinate directly with each other through DOM and browser APIs.

### Stack Management Challenges

When multiple overlays are open simultaneously across different JavaScript bundles (micro frontends), coordination becomes critical:

1. **Visual stacking (z-index)**: Which overlay appears on top - cannot rely on render order
2. **Cross-bundle communication**: No shared JavaScript memory - must coordinate through DOM/browser APIs
3. **Peer-to-peer coordination**: No host application - MFEs must coordinate directly with each other
4. **Version compatibility**: Different MFEs may use different Bento versions (v1, v2, v3) simultaneously
5. **Priority system**: Priority-based stacking where higher priority overlays appear on top
6. **Keyboard focus**: Which overlay receives focus based on priority, not just z-index
7. **ARIA attributes**: Proper `aria-modal` and `aria-hidden` attributes coordinated across bundles

### UX Patterns for Multiple Overlays

**Pattern 1: Sequential Dismissal**
- Closing topmost overlay returns focus to next overlay below
- Users dismiss overlays in reverse order of appearance
- Each overlay maintains its own state independently
- **Most common pattern** - used by default

**Pattern 2: Cascade Dismissal**
- Closing a parent overlay closes all child overlays
- Useful for modal dialogs opened from other modals
- Requires explicit parent-child relationships via props
- **Optional pattern** - requires custom implementation

**Pattern 3: Independent Stacks**
- Overlays of different types (modal, tooltip, popover) stack independently
- Each type has its own z-index base
- Focus remains within each type's stack
- **Automatic** - handled by overlay-stack primitive

### Implementation

The `@bento/overlay-stack` primitive provides **peer-to-peer cross-bundle orchestration**:

- **Peer-to-peer coordination**: MFEs coordinate directly - no host application required
- **Leader election**: One MFE instance becomes leader and coordinates stacking (based on version)
- **Simple communication**: Uses Custom Events only - no persistence needed
- **In-memory state**: All state is ephemeral and resets on page refresh (correct behavior)
- **Version-aware**: Handles different Bento versions (v1, v2, v3) running simultaneously
- **Priority system**: Priority-based stacking where higher priority overlays appear on top
- **Automatic z-index calculation**: Based on priority, type, and timestamp (not just insert order)
- **Focus management**: Only topmost overlay traps focus, coordinated across bundles
- **ARIA coordination**: Only topmost modal has `aria-modal="true"`, lower modals get `aria-hidden="true"`
- **Universal pattern**: Works for any competing UI layer (overlays, toasts, growls, tours)

**Key difference**: This isn't just React state management - it's a peer-to-peer orchestration system that works when MFEs can't share JavaScript memory and don't have a host application.

See `overlay-stack-primitive.mdx` for detailed implementation including micro frontend scenarios and version compatibility.

### Nested Overlays (Dropdowns in Modals) - CSS Variables!

When a dropdown or popover is rendered inside a modal, it's **automatically nested** using CSS custom properties:

```tsx
import { Overlay } from '@bento/overlay';
import { Select } from '@bento/select';

function ModalWithDropdown() {
  return (
    <Overlay>
      {/* Overlay sets --layer-z-index: 4000 */}
      <Container slot="content">
        <Text>Modal Content</Text>

        {/* Dropdown uses calc(var(--layer-z-index) + 1) */}
        <Select>
          <Dropdown /> {/* Automatically z-index: 4001 */}
        </Select>
      </Container>
    </Overlay>
  );
}
```

**How it works:**
- `Overlay` sets CSS variable `--layer-z-index` on its element
- Nested layers detect parent CSS variable via `getComputedStyle`
- Nested layers use `calc(var(--layer-z-index) + 1)` for relative z-index
- CSS inheritance handles everything automatically
- **Zero configuration** - standard web platform feature!

**Key points:**
- ✅ Uses CSS custom properties (standard web platform)
- ✅ No React Context or manual tracking needed
- ✅ Works across all frameworks (React, Vue, vanilla JS)
- ✅ Automatic inheritance through DOM tree
- ✅ Can force global with `forceGlobal={true}` if needed

### Example: Multiple Overlays (Same Bundle)

```tsx
import { OverlayStackProvider } from '@bento/overlay-stack';
import { Overlay } from '@bento/overlay';
import { Button } from '@bento/button';
import { Backdrop } from '@bento/backdrop';
import { Container } from '@bento/container';
import { Text } from '@bento/text';
import { DismissButton } from '@bento/dismiss-button';
import { useState } from 'react';

function MultipleOverlays() {
  const [firstOpen, setFirstOpen] = useState(false);
  const [secondOpen, setSecondOpen] = useState(false);

  return (
    <OverlayStackProvider>
      <Button onClick={() => setFirstOpen(true)}>Open First Overlay</Button>

      <Overlay isOpen={firstOpen} onOpenChange={setFirstOpen}>
        <Button slot="trigger">Open First</Button>
        <Backdrop slot="backdrop" />
        <Container slot="content">
          <Text>First Overlay</Text>
          <Button onClick={() => setSecondOpen(true)}>Open Second Overlay</Button>
          <DismissButton slot="close">Close First</DismissButton>
        </Container>
      </Overlay>

      <Overlay isOpen={secondOpen} onOpenChange={setSecondOpen}>
        <Backdrop slot="backdrop" />
        <Container slot="content">
          <Text>Second Overlay</Text>
          <Text>This overlay appears above the first one.</Text>
          <DismissButton slot="close">Close Second</DismissButton>
        </Container>
      </Overlay>
    </OverlayStackProvider>
  );
}
```

### Example: Cross-Bundle Orchestration (Micro Frontends - No Host)

```tsx
// MFE A (Bento v2): Product Tour
import { useOrchestratedLayer } from '@bento/overlay-stack';
import { Overlay } from '@bento/overlay';

function ProductTourMFE() {
  const { ref, zIndex, isTopmost } = useOrchestratedLayer({
    type: 'tour',
    priority: 250, // Low priority (0-1000 scale)
    isOpen: true,
    mfeId: 'product-tour-mfe',
    version: '2.1.0', // Bento version
  });

  return (
    <Overlay isOpen={true}>
      <Container slot="content" ref={ref} style={{ zIndex }}>
        Product Tour (may be behind cookie consent)
      </Container>
    </Overlay>
  );
}

// MFE B (Bento v1): Legal/Cookie Consent
function CookieConsentMFE() {
  const { ref, zIndex, isTopmost } = useOrchestratedLayer({
    type: 'modal',
    priority: 1000, // Critical priority (highest)
    isOpen: true,
    mfeId: 'legal-mfe',
    version: '1.2.0', // Older Bento version, still works
  });

  // Even though v1 is older, it gets topmost due to priority
  // v2 instance becomes leader (higher version) and calculates stack
  return (
    <Overlay isOpen={true}>
      <Container
        slot="content"
        ref={ref}
        style={{ zIndex }}
        aria-modal={isTopmost ? 'true' : undefined}
      >
        Cookie Consent (always on top due to critical priority)
      </Container>
    </Overlay>
  );
}
```

The overlay-stack primitive automatically:
- **Peer-to-peer coordination**: MFEs coordinate directly, no host needed
- **Leader election**: Highest version MFE becomes leader and coordinates
- **Version compatibility**: Different Bento versions (v1, v2, v3) interoperate
- **Priority-based stacking**: Higher priority overlays appear on top (cookie consent wins)
- **Sets `aria-modal="true"`** only on topmost modal (cookie consent)
- **Sets `aria-hidden="true"`** on lower overlays (product tour)
- **Works even when bundles load in different orders**
- **Handles competing toasts/growls from different MFEs and versions**

- **ARIA roles**: Automatically applies appropriate roles (`role="dialog"` for modals)

## Competitive Research

We conducted comprehensive research across major component libraries:

### React Aria

React Aria provides hooks for building accessible overlays:
- **useModalOverlay**: Manages ARIA attributes and keyboard interactions
- **useOverlay**: Core overlay behavior
- **useOverlayTrigger**: Connects triggers to overlays

**Strengths:**
- Exceptional accessibility
- Robust focus management
- Strong mobile support

**Adopted as foundation**: We use React Aria as our base, enhancing with Bento's slot-based composition model.

### Radix UI

Radix UI offers a highly composable dialog component with compound components:

```tsx
<Dialog.Root>
  <Dialog.Trigger />
  <Dialog.Portal>
    <Dialog.Overlay />
    <Dialog.Content>
      <Dialog.Title />
      <Dialog.Description />
      <Dialog.Close />
    </Dialog.Content>
  </Dialog.Portal>
</Dialog.Root>
```

**Differences:**
- Radix uses compound components (forbidden in Bento)
- Bento uses slot-based system (consumers provide components as children)

**Insights adopted:**
- Data attribute-based state for styling
- Portal rendering pattern

### Ark UI / Park UI

Ark UI (headless) and Park UI (styled) provide similar patterns to Radix:

**Differences:**
- Also use compound components
- Bento's slot system is more flexible

**Insights adopted:**
- Separation of concerns (backdrop, content, trigger)

### Basic usage

```tsx

}
```

### Multiple overlays with stacking

```tsx
import { OverlayStackProvider } from '@bento/overlay-stack';

function MultipleOverlays() {
  const [firstOpen, setFirstOpen] = useState(false);
  const [secondOpen, setSecondOpen] = useState(false);

  return (
    <OverlayStackProvider>
      <Button onClick={() => setFirstOpen(true)}>Open First Overlay</Button>

      <Overlay isOpen={firstOpen} onOpenChange={setFirstOpen}>
        <Backdrop slot="backdrop" />
        <Container slot="content">
          <Text>First Overlay</Text>
          <Button onClick={() => setSecondOpen(true)}>Open Second Overlay</Button>
          <DismissButton slot="close">Close First</DismissButton>
        </Container>
      </Overlay>

      <Overlay isOpen={secondOpen} onOpenChange={setSecondOpen}>
        <Backdrop slot="backdrop" />
        <Container slot="content">
          <Text>Second Overlay</Text>
          <Text>This overlay appears above the first one.</Text>
          <DismissButton slot="close">Close Second</DismissButton>
        </Container>
      </Overlay>
    </OverlayStackProvider>
  );
}
```

The overlay-stack primitive automatically handles:
- Higher z-index for second overlay (4010 vs 4000)
- Focus trapping in topmost overlay
- ARIA attributes (only topmost has aria-modal="true")
- Focus restoration when second closes
