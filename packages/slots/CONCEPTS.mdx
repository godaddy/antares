# Slots

Slots gives you full control over every part of the component. It's the ultimate
expression of freedom in a component-based architecture. Every component that is
build with the `@bento/slots` now accepts 2 additional props:

<ArgTypes of={Stories.SlotsAPI} />

> **IMPORTANT**
>
> Before you introduce `slots` to a component reach out to your design system
> team. Overrides are an indication that the component you're using is no
> longer following the design system guidelines or the design that you're
> trying to implement is not covered by the design system.
>
> Components are used to enforce consistency across the application. Reaching
> out to your design system team will help to ensure that the component you're
> using is still following the design system guidelines.
>
> If you are fixing a bug, consider contributing the fix back to the design
> system. This will help to ensure that the component is fixed for everyone.

The `slots` prop accepts an object as a value. The key of the object should be
the name of the slot that you want to modify. The value of the object should be
an object or function that contains the modifications that you want to make to
the slot.

## Assigning `slot`'s to components

Note that this only needed when you are building your own custom components on
top of Bento that should also support slots. If you are developing applications
then this is already done for you in the components that you consume.

The slots implementation needs to know which component should receive the
modifications, this is done by assigning the `slot` prop. The `slot` prop can be
assigned to any component that is built with the `@bento/slots` package. The
`slot` accepts a string as a value that represents the name of the slot.

The example component below will be used througout the documentation to
illustrate the usage of slots.

While it is possible to use assign the same name `slot` to multiple (different)
components, it is recommended to use unique names to allow individual targeting
of each component. When authoring components the names of the slots should be
documented in the component's documentation so the consumers know which slots
are available.

## Namespace for component composition

A unique feature of our slots implementation is the ability to drill down
components. When a `slot` is assigned to a component it becomes part of the
namespace of the component. This namespace can be used to drill down the
component tree and access the slots of the children.

The namespace is a dot-separated string that represents the path to the slot.
To illustrate this, consider the following example where we created a fictional
`Search` component that is composed with an existing `Button` component. This
`Button` component has a slot named `icon` that we want to customize.

```jsx
// button component
<Button>
   <Text slot="label">{ children }</Text>
   <Icon slot="icon" name="arrow-right" />
</Button>

// search component
<Search>
   <Label slot="label">
      Search
      <Icon slot="icon" name="search" />
   </Label>
   <Input slot="search" />
   <Button slot="submit">Submit</Button>
</Search>
```

As the namespace is composed of each of the assigned `slot` name, we can
customize the `icon` slot of the `Button` component by targeting the
`submit.icon` namespace. The namespace should be supplied as key and the
required modifications as value as illustrated below:

```jsx
<Search slots={{ 'submit.icon': { name: 'search' } }} />
```

## Introducing new props

When an object is supplied as a value to a slot, the object will be merged with
the props of the component. This allows you to introduce or override props of
the component.

## Event Handlers via Slots

Event handlers can be passed through the slot system to child components, enabling flexible prop distribution without explicit prop drilling. This is a core Bento architectural pattern used in coordinator components like `Select`, `Menu`, and `Dialog`.

### How Event Handlers Flow

When a coordinator component passes an event handler via slots, the handler flows through the Bento context system to properly wrapped child components:

```tsx
// Coordinator passes event handler via slots
<Container
  slots={{
    popover: { isOpen: true, onClose: handleClose }
  }}
>
  <CustomPopover slot="popover">
    <ListBox>...</ListBox>
  </CustomPopover>
</Container>
```

### Receiving Event Handlers in Child Components

Child components must use `withSlots` and `useProps` to receive event handlers from the slot system:

```tsx
import { withSlots } from '@bento/slots';
import { useProps } from '@bento/use-props';

const CustomPopover = withSlots('CustomPopover', function CustomPopover(args) {
  const { props } = useProps(args);
  const { onClose, isOpen, children, ...rest } = props;
  
  // onClose is extracted from the slot context
  return (
    <div {...rest} style={{ display: isOpen ? 'block' : 'none' }}>
      <button onClick={onClose}>Close</button>
      {children}
    </div>
  );
});
```

### Event Handler Detection

Event handlers are automatically identified by the `useProps` hook using the pattern `/^on[A-Z]/`. Any prop that starts with "on" followed by an uppercase letter is treated as an event handler:

- ✅ `onClick`, `onChange`, `onClose`, `onSubmit` - Detected as event handlers
- ❌ `onclick`, `onclose` - Not detected (lowercase after "on")
- ❌ `handleClick`, `closeHandler` - Not detected (doesn't start with "on")

### Why This Pattern?

**Traditional prop drilling** requires manually passing props through every layer:

```tsx
// Without slots - requires explicit prop passing
<Select>
  <Popover onClose={handleClose}>
    <div>
      <ListBox>...</ListBox>
    </div>
  </Popover>
</Select>
```

**Slot-based event handlers** enable flexible composition:

```tsx
// With slots - handler flows through context
<Select slots={{ popover: { onClose: handleClose } }}>
  {/* User can nest any component structure */}
  <div>
    <div>
      <CustomPopover slot="popover">
        <ListBox>...</ListBox>
      </CustomPopover>
    </div>
  </div>
</Select>
```

**Benefits:**
- No prop drilling through intermediate components
- Supports arbitrary component nesting
- Coordinator doesn't need to know child component structure
- Type-safe via TypeScript slot interfaces

**Requirements:**
- Child components must use `withSlots` HOC
- Child components must use `useProps` to extract handlers
- Proper slot assignment via `slot` prop

### Common Pitfalls

**Forgetting `withSlots`:**
```tsx
// ❌ Won't receive event handlers
const MyPopover = (props) => {
  const { onClose } = props; // undefined!
  return <div onClick={onClose}>...</div>;
};

// ✅ Receives event handlers
const MyPopover = withSlots('MyPopover', function MyPopover(args) {
  const { props } = useProps(args);
  const { onClose } = props; // ✅ extracted from context
  return <div onClick={onClose}>...</div>;
});
```

**Not using `useProps`:**
```tsx
// ❌ Event handlers not extracted
const MyPopover = withSlots('MyPopover', function MyPopover(props) {
  const { onClose } = props; // undefined!
  return <div>...</div>;
});

// ✅ Event handlers extracted
const MyPopover = withSlots('MyPopover', function MyPopover(args) {
  const { props } = useProps(args); // ✅ extracts from context
  const { onClose } = props;
  return <div>...</div>;
});
```

**Incorrect slot assignment:**
```tsx
// ❌ Slot name doesn't match
<Container slots={{ popover: { onClose: handleClose } }}>
  <MyComponent slot="overlay">...</MyComponent> {/* Wrong slot name! */}
</Container>

// ✅ Slot names match
<Container slots={{ popover: { onClose: handleClose } }}>
  <MyComponent slot="popover">...</MyComponent> {/* ✅ Matches */}
</Container>
```

### Real-World Example

The `@bento/select` primitive uses this pattern to pass the `onClose` handler to custom popover components:

```tsx
import { Select, SelectOption } from '@bento/select';
import { withSlots } from '@bento/slots';
import { useProps } from '@bento/use-props';

// Custom popover that receives onClose from Select
const CustomPopover = withSlots('CustomPopover', function CustomPopover(args) {
  const { props } = useProps(args);
  const { onClose, isOpen, children } = props;
  
  return isOpen ? (
    <div className="custom-popover">
      <button onClick={onClose}>✕</button>
      {children}
    </div>
  ) : null;
});

// Select automatically passes onClose to the popover slot
<Select>
  <Button slot="trigger">Select Option</Button>
  <CustomPopover slot="popover">
    <ListBox slot="listbox">
      <SelectOption value="1">Option 1</SelectOption>
    </ListBox>
  </CustomPopover>
</Select>
```

See the [`@bento/select` documentation](../../docs/pdrs/select-primitive.mdx#slot-system-event-handler-pattern) for more details on this pattern in action.

## Modify children

When a function (`renderProp`) is supplied as a value to a slot, the function
can be used to replace or modify the children of a component. This allows you to
affect the structure of the component.

The function will receive the `props`, `original` and `previous` as arguments and should
return the content that should be rendered. The `props` argument is an object
that contains the original props of the component. The `original` argument is
the original content of the slot. The `previous` argument is an array of the previous slots if any.

## Best practices

As the `slots` prop accepts an object, it is important to ensure that the
object is memoized to prevent unnecessary re-renders. If the modifications
you are making to the component are static, you can declare the object outside
of the component scope and pass it as a prop. This ensures that the object is
only created once and the prop is not considered a new prop on each render:

```jsx
const slots = {
  'submit.icon': { id: 'my-custom-id' },
};

function YourExampleComponent() {
  return <Search slots={slots} />;
}
```

If your object references variables declared in the component scope you can use
the `React.useMemo` hook. The `React.useMemo` hook caches the object between
re-renders. Given that your object references variables declared in the
components function scope you must include them in the dependencies array:
