# Props and state as `data-` attributes

Components often have different UI state variations. Bento exposes these
variations as `data-` attributes to allow for dynamic styling and conditional
rendering based on the component's state. When we are talking about UI
states, we are referring to the different visual states that a component can be
in, e.g.:

- **Interactive states**: User interactions like `hovered`, `focused`, and `pressed`.
- **Animation states**: `entering` and `exiting` animations.
- **Internal state**: `loading`, `disabled`, `invalid`, and `valid` states etc.
- **Props**: Any property indicating a visual difference in the component.

All `data-` attributes supported by each component are listed as part of their
API documentation.

## Returning control

By exposing the props, interactions, and state of a component as `data-`
attributes, we can return the styling control to CSS. This allows for
dynamic, conditional layouts and designs to be composed based on attribute
selectors.

To give an example, let's say we have a button component that has a `variant`
prop. Instead of having the component manage the styles or conditionally render
different classes, we expose the `variant` as a `data-` attribute and let CSS
handle the rest.

```jsx
<ButtonExample className="my-button" variant="primary">Hello World</ButtonExample>
<button class="my-button" data-variant="primary">Hello World</button>
```

We can target the introduced props using CSS attribute selectors to apply
different styles based on selected `variant` and interaction states. In the
the example below, we are changing the background color of the button to a darker
shade of blue when the button is pressed:

```css
.my-button[data-variant="primary"] {
  background-color: blue;

  &[data-pressed] {
    background-color: darkblue;
  }
}
```

## Debugging

The more information available in the DOM, the easier it will be to
debug your components without needing specialized tools or plugins. The markup
will tell you exactly what state your Component is in.

```html
<a href="#" class="button_6j83f" data-loading="true" data-disabled="true" data-variant="primary">
  Fetching data...
</a>
```

As we can see from the example HTML output above, we're able to deduct the
following:

- `data-loading` - The button is currently in a loading state.
- `data-disabled` - The button is disabled.
- `data-variant` - The button has a primary variant.

## Best Practices

The `useDataAttributes` returns a memorized object. You do not need to wrap
the returned object in a `useMemo` hook.

### Which props should be exposed?

Not all props should be exposed as `data-` attributes. Only introduce the
`data-` attributes for properties that do not end up in the resulting HTML
of the component. A `class` attribute and a `data-class` attribute
would be redundant and confusing.

This doesn't just apply to props that are rendered as part of the root element
but also to properties that end up as nested elements. If a property is rendered
there is no need to introduce a `data-` attribute in any way, shape, or form.

To give an example of this, our `@bento/illustration` component allows you to
specify the `title` for accessibility purposes. This property is introduced
as part child of the `svg` element as the `<title>` element. As the property is
already exposed in the DOM and can be used for targeting using the `:has`
CSS selector:

```css
svg:has(title) {
  border: 1px solid red;
}
```

A good example of a prop that should be exposed as a `data-` attribute is
the `icon` prop from `@bento/icon`. The `icon` prop is the icon's name
that should be rendered. While it does introduce content as part of its
`children`, it's impossible to target using CSS.

```jsx
<Icon className="my-className-here" icon="close" />
```

By introducing a `data-icon` attribute, we can target the icon using CSS:

```css
.my-className-here[data-icon="close"] {
  /* We can use color because icons use currentColor by default */
  color: red;
}
```

### Default to existing HTML attributes over data attributes

When possible, default to using existing HTML attributes when conveying
state to developers. When working with form components, we might want to convey
the `invalid` and `valid` states to developers. We can introduce `data-valid`
and `data-invalid` as attributes to the component, but there are dedicated ARIA
attributes for this purpose. In these cases, we would assign `aria-invalid`
and `aria-valid` as attributes for the component.

### Naming conventions

To have a consistent naming framework for our `data-` attributes, we
are defaulting to the naming conventions that are used in React ARIA to convey
component states.

When there is no matching state in React ARIA or existing HTML attribute that
can use, we default to the following naming convention:

- `data-loading={true|undefined}` - Indicates that the element is in a loading state.

When authoring your custom components, it's recommended that you follow the
Use the same naming convention to ensure consistency across your application.
